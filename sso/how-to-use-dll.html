# Dll 动态链接库配置

在用 webpack 打包的时候，对于一些不经常更新的第三方库，比如 element-ui，vue 等，我们希望能和自己的代码分离开

比较常见的方法有三种

  1. 直接使用 cdn
  2. 使用 CommonsChunkPlugin
  3. 使用 DLLPlugin

上面三种方法各有优缺点，这里我们选用第三种，dll 模式，即把第三方库单独打包成，从而源码打包时可以忽略对应模块，提高打包效率和缩短打包时间。

# 新建 DLL 配置文件

dll 文件是把一些第三方库单独打包，所以我们也要单独做个配置来执行他们。我们在项目的 webpack 配置文件夹build中添加一个dll.config.js文件(如果是其他项目，写法也类似)，内容如下：

// build/dll.config.js
const webpack = require('webpack');

// 设置动态链接库的名称
const library = '[name]_lib';
const path = require('path');

module.exports = {
	entry: {
		// 设置入口已经需要单独打包的模块
		vendors: ['vue', 'vue-router', 'element-ui'],
	},

	output: {
		filename: '[name].dll.js', // 打包后的文件名称
		path: path.resolve(__dirname, '../dist'), // 打包后的文件位置
		library, // 动态链接库名称
	},

	plugins: [
		// 使用webpack内置的DllPlugin
		new webpack.DllPlugin({
			// 生成dll信息的文件
			path: path.join(__dirname, '../dist/[name]-manifest.json'),
			// dll名称，记住要和output的library属性保持一模一样
			name: library,
		}),
	],
};

# 添加 Script 命令

添加了 build/dll.config.js 后，程序并不会自动执行，这时候我们需要在package.json中添加一条新命令，命令如下

  "scripts": {
    "dll": "webpack --config build/dll.config.js",
  },

显然，我们添加了一个 dll 命令，该命令使用 webpack 调用，传入的配置参数是 build/dll.config.js 导出的结果。

# 生成 DLL

这时候我们执行下面的命令

$ npm run dll

执行完成后我们可以发现,它在dist文件夹中生成了下面两个文件

  • vendors-manifest.json
  • vendors.dll.js

通过查看代码发现，vendors-manifest.json 大致信息内容如下

{
	"name": "vendors_lib",
	"content": {
		"./node_modules/_vue@2.6.10@vue/dist/vue.runtime.esm.js": {
			"id": 2,
			"meta": { "harmonyModule": true },
			"exports": ["default"]
		}
	}
}

发现vendors-manifest.json中包含了那些需要单独打包模块的位置，以及发现里面的 name 属性是刚刚配置的 vendors_lib。

然后我们再去看 vendors.dll.js，发现它里面包含了全部需要单独单独打包的模块。

# 修改 Prod 生产配置文件

前面我们再 dist 文件夹下生成了两个文件，如果不做其他修改，打包的时候依旧还是会正常把那些第三方打进去的，所以我们还需要在 prod 生产配置下做多个处理

在生产配置下，我们需要使用DllReferencePlugin模块，它可以告诉webpack，哪些模块我已经打包了，不需要再次打包（有点类似于 webpack 下的 externals，但这里不需要再使用 externals 了）

值得注意的时，有两个DllReferencePlugin模块

  • const DllReferencePlugin = require('webpack/lib/DllReferencePlugin');

  • new webpack.DllReferencePlugin

这两个模块功能和配置一样，但适用于不同的 webpack 版本,如果没有用对，是不会有效果的（也就是说打包还是一样大）

如果是 webpack 2.x，需要使用第一种，如果是 webpack4.x，则两个都可以使用，具体看 webpack 版本

比如本项目使用的是第一种，第二种不生效（已升级到webpack4.x） 下面是具体代码

如果使用第一种，需要单独导入

// build/prod.config.js 导入dll插件
const DllReferencePlugin = require('webpack/lib/DllReferencePlugin');

如果是第二种，webpack 已经内置，可以直接使用

new webpack.DllReferencePlugin(options);

确定DllReferencePlugin引入方式后，我们在plugins中添加 dll 配置

// build/prod.config.js
plugins: [
	// ...
	new DllReferencePlugin({
		// 导入dll的依赖配置，webpack会自动忽略相关模块
		manifest: require('../dist/vendors-manifest.json'),
	}),
	// 或
	//new webpack.DllReferencePlugin({
	//	manifest: require('../dist/vendors-manifest.json'),
	//}),
	// ...
];

还需要注意的是，一般来说我们在打包前都会先清空dist文件夹，这也就会把我们之前创建的 dll 文件给误删除了,所以我们需要对 dll 文件做特殊处理，继续保留。

如果是使用clean-webpack-plugin进行文件清除的，那么可以做下面的配置

const { CleanWebpackPlugin } = require('clean-webpack-plugin');

plugins: [
	// ...
	new CleanWebpackPlugin({
		// default: ['**/*']
		cleanOnceBeforeBuildPatterns: ['**/*', '!*.dll.js', '!*-manifest.json'],
	}), // 保留前面生成的dll文件
	// ...
];

这时候我们再去进行打包

$ npm run build

可以发现打包出来的文件明显变小

如果发现没有生效（也就是没有变小），修改成另一个下 DllReferencePlugin 模块即可（不要尝试用修改 externals 的方式去忽略）

# 引入 DLL 文件

既然已经生成 dll 文件了，而且打包出来的文件也没有第三方模块，那么直接运行肯定是报错的，所以我们需要单独引入生成的 dll 文件

引入方式有两种，第一种是直接通过 script 的方式引入，这个写法简单，但不方便维护，我们介绍第二种自动引入的方式。

首先我们分别在 dev 和 prod 配置的 htmlWebpackPlugin 插件上添加一个 dll 数组，如：

new htmlWebpackPlugin({
	// ...
	dll: [`${prodConfig.output.publicPath}/vendors.dll.js`], // 自行拼接好路径
	// ...
});

配置好一个可访问的 url 后，我们再去到 index.html 中动态引入此文件

<!-- index.html -->
<!DOCTYPE html>
<html lang="zh-CH">
	<head></head>
	<body>
		<% htmlWebpackPlugin.options.dll.forEach((item)=>{ %>
		<script type="text/javascript" src="<%= item %>"></script>
		<% }) %>
	</body>
</html>

其中上面语句是 ejs 语句。

最后在打包的时候会自动加上 dll 文件。
