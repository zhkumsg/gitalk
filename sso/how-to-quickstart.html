# 快速开始

单点登录（Single Sign On），简称为 SSO，是目前比较流行的企业业务整合的解决方案之一。SSO 的定义是在多个应用系统中，用户只需要登录一次就可以访问所有相互信任的应用系统。

# Installation

# 生产环境

$ npm install
$ npm run start

# 开发环境

$ npm install
$ npm run:node
$ npm run:vue

启动后将在 http://localhost:3000 启动 node 服务，在 http://localhost:3002 启动 vue 服务

其中 node 服务的静态资源依赖于打包后的 vue 项目。

可单独启动 node 服务而不启动 vue 项目

# 单点登录

# 注册应用

在使用前首先在单点系统上注册应用，注册成功后会得到 appid 和 appsecret ，将来需要用这两个参数做身份认证。

# 页面跳转

创建应用后，在需要登录的时候跳转到单点系统的授权页面，如 http://localhost:3000/api/oauth/${appid}/${callback_url} 。

跳转时需要携带下面的参数

参数 说明
appid 应用标识, 在应用类别中可查看
callback_url 回调地址，经过 UrlEncode 编码后能正确访问的 url 地址

eg:

应用标识为： 1566383195899 、回调地址为： https://www.baidu.com , 此时正确的跳转地址如下

http://localhost:3000/oauth/1566383195899/https%3a%2f%2fwww.baidu.com

# 登录回调

在单点系统成功登录后，会按照传入的 callback_url 进行重定向，回调时会自动带上 key 信息，如

https://www.baidu.com?key=eyJpdiI6Ik1sSUJaK3RcL3FCVzVcLzl1

此时页面带参跳转回应用的回调页面。

  • 如果是已登录状态时，其他子应用进入会瞬间完成回调授权，不会看到登录页面

PS：key 的有效期为 5 分钟

# key 转 token

在应用中获取用户信息前必须通过 /api/key2token 接口形式从单点系统获取相关 token，需要传入以下参数

参数 说明
appid 应用标识
appsecret 应用私钥
key 授权回调后的参数

接口要求如下：

  • 请求方式为 post

  • 数据格式为 application/json

返回结果如例：

{
	"status": true,
	"data": 'eyJpdiI6Ik1sSUJaK3RcL3FCVzVcLzl1eyJpdiI6Ik1sSUJaK3RcL3FCVzVcLzl1eyJpdiI6Ik1sSUJaK3RcL3FCVzVcLzl1eyJpdiI6Ik1sSUJaK3RcL3FCVzVcLzl1eyJpdiI6Ik1sSUJaK3RcL3FCVzVcLzl1eyJpdiI6Ik1sSUJaK3RcL3FCVzVcLzl1eyJpdiI6Ik1sSUJaK3RcL3FCVzVcLzl1',
	"msg": "",
}

其中 status 为 true 时代表接口正常, data 代表 token 信息，后续与 sso 的交互都需要携带

key 一次有效，只能使用一次，且五分钟内有效

# 用户认证

在应用中获取用户信息时必须通过 /api/token2userinfo 接口形式从单点系统获取，需要传入以下参数

参数 说明
appid 应用标识
appsecret 应用私钥
token 通过 key 换取的 token

接口要求如下：

  • 请求方式为 post

  • 数据格式为 application/json

返回结果如例：

{
	"status": true,
	"data": {
		"id": "admin",
		"name": "管理员",
		"avatar": "https://c-ssl.duitang.com/uploads/item/201712/22/20171222223729_d8HCB.jpeg"
	},
	"msg": "",
}

其中 status 为 true 时代表接口正常

token 60 分钟内有效

# 单点注销

单点登录自然也会有单点注销，在一个子系统中注销，所有子系统的会话都将被销毁。

下面是具体的应用场景，包括登录和注销，可按实际场景选择。

建议使用 NODE-SSO-CONNECT 模块，一键适配全部场景。

# 域名说明

本机上配置多个域名地址，其中 sso.test.com 用于做单点服务， a/b/c/d 分别代表不同的子应用。

具体设置，[点我](http://localhost:3000/admin/doc/1. Host%E9%85%8D%E7%BD%AE.md)

# 应用场景

# 1 同一顶级域名

a.test.com 、b.test.com、sso.test.com

描述：N 个系统，但是 一级域名 是一致的 。如果不懂 一级域名 、 二级域名 是什么意思先查看这篇文章。《二级域名是什么》。这个案例实现相对简单。

原理：其实质这里就是利用了 二级域名 写 一级域名 的 Cookie 。

优点：轻量级、可插拔、效率非常高。

缺点：局限性限于同一域名。

# 2 不同顶级域名

a.test.com 、a.debug.com、a.dev.com
  • a：通过 iframe
  • b：全部请求都过用户中心

# 3 不限域名

  • 通过 sso 主动推送方式
