# 异常捕获与上报

异常捕获与上报是异常监控的前提，了解并做好了异常数据的收集和分析才能实现一个完善的错误响应和处理机制，最终达成数据可视化, 后面将介绍如何在已有项目上集成 Sentry

# 前 言

正所谓百密一疏，一个经过了大量测试及联调的项目在有些时候还是会有十分隐蔽的 bug 存在，这种复杂而又不可预见性的问题唯有通过完善的监控机制才能有效的减少其带来的损失，因此对于直面用户的前端而言，异常捕获与上报是至关重要的。

对服务端来说还好, 可以通过日志的形式来记录信息, 而对于前端而言, 异常都是发生在客户端的, 必须得通过上报的形式才可以被程序员所得知

为了让前端也能和后端一样，需要将线上的 JavaScript 代码监控起来，当用户端浏览器出现异前端第一时间被通知到(甚至邮件实时推送)

# 采集哪些数据

主要原则就是避开用户敏感字段，采集浏览器版本、操作系统版本、报错的 msg 信息等

# 如何采集错误

前端错误大体上可以分成两类，

  • 代码执行的错误
  • 资源加载的错误(包括但不限于 http 异常)

下面是具体的捕获方式

  • 使用 try catch
  • 使用 window.onerror
  • 框架集成的错误中间件

# 如何上报错误

  • 使用 ajax 上报
  • 使用 image 上报

一般来说，大厂都是采用利用 image 对象的方式上报错误的；使用图片发送 get 请求，上报信息，由于浏览器对图片有缓存，同样的请求，图片只会发送一次，避免重复上报。

# 选择监控方法

  • 自行搭建
  • Sentry
  • Fundebug
  • Bugsnag
  • Better JS

由于需要监听 Vue 项目和 nodejs 项目, 且监听服务需要跑在自己的服务器上, 只有自行搭建, Sentry 和 Better JS 三个可选, 但对于 vue 项目, 代码已经经过压缩混淆了, 所以监控方案需要具备解析 Map 的功能, 出于开发成本和集成难度的考虑, 最终决定使用 Sentry

Sentry 官网

# 搭建 Sentry 服务

Sentry 是一个开源的实时错误追踪系统，可以帮助开发者实时监控并修复异常问题。它主要专注于持续集成、提高效率并且提升用户体验。Sentry 分为服务端和客户端 SDK，前者可以直接使用它家提供的在线服务，也可以本地自行搭建；后者提供了对多种主流语言和框架的支持，包括 React、Angular、Node、Django、RoR、PHP、Laravel、Android、. NET、JAVA 等。同时它可提供了和其他流行服务集成的方案，例如 GitHub、GitLab、bitbuck、heroku、slack、Trello 等。目前公司的项目也都在逐步应用上 Sentry 进行错误日志管理。

Sentry 本身是基于 Django 开发的，而且也依赖到其他的如 Postgresql、 Redis 等组件，所以一般有两种途径进行安装：

  • 通过 Docker
  • 用 Python 搭建

下面介绍如何使用 docker 安装

# 克隆 Github 项目

github 上有个开源项目用于部署 Sentry, 首先我们需要先把它克隆到本地

$ git clone https://github.com/getsentry/onpremise.git

如果没有安装 git 环境, 也可以下载项目到本地 点击查看

克隆完成后, 我们进入 onpremise 文件夹

$ cd onpremise

项目结构

┌── config.yml
├── Dockerfile # docker 配置
├── sentry.conf.py # sentry配置
├── test.sh # 测试脚本
├── install.sh # 安装脚本
├── Makefile # 编辑
├── requirements.txt
└── docker-compose.yml # docker-compose相关

# 搭建 docker 环境

node 和 npm 环境搭建不再阐述

因为我们使用简单的 docker 一键部署方式, 所谓我们当然是需要配置好 docker 环境的, 下面以 Ubuntu 为例

$ sudo apt install docker.io docker-compose

如果没有 apt 包管理工具,也可以使用 yum 下载

安装 docker 相关服务, docker.io 和 docker-compose

# 执行安装脚本

这就是我们前面提到的 onpremise/install.sh , 它是一段 shell 脚本, 执行后会自动安装 Sentry

$ sudo ./install.sh

正常执行后, 等待一段时间, 会自动安装服务

在安装的后部分, 会提示输入超级用户的 账号和密码(以及确认密码) , 这个账号将作为 Sentry 的超级管理员

# 启动 Sentry

安装完成后, 按照提示运行

$ sudo docker-compose up -d

如无意外, 将在 9000 端口启动 Sentry 服务 点击打开

# 创建项目

成功登录 Sentry 后, 在 Project 菜单中找到 add new 按钮, 选择创建一个项目

这时候按照提示, 选择项目目标语言, 最后会拿到一段 dsn 字符串, 如

http://1b036df22e114db78f99770cf9be29b0@localhost:9000/3

按照页面提示, 再去代码中添加相关内容即可完成异常监控

# nodejs 项目集成

按照官网文档, 我们只需要作两步就可以接入 Sentry 了

  1. 安装依赖
$ npm install @sentry/node@5.5.0 --save
  1. 初始化服务
// app.js
const Sentry = require('@sentry/node');
Sentry.init({
	dsn: 'http://1b036df22e114db78f99770cf9be29b0@localhost:9000/3',
});

这就完成了 Sentry 的接入(注意这里的 dsn)

如果是 koa 项目, 还可以在中间件中进一步监听异常

// app.js
const app = new Koa();

app.on('error', (err, ctx) => {
	Sentry.withScope(scope => {
		scope.addEventProcessor(event => Sentry.Handlers.parseRequest(event, ctx.request));
		Sentry.captureException(err);
	});
});

# vue 项目集成

同样的, 在 vue 下也是两步就可以完成集成, 不过需要注意的是, 需要考虑是否需要新建多一个 Sentry 项目, 如果希望 nodejs 的异常与 vuejs 的异常分离的话, 可以按照前面创建项目的教程新创一个 vuejs 的项目

  1. 安装依赖
$ npm install @sentry/browser  @sentry/integrations --save
  1. 初始化 Sentry
// main.js
import Vue from 'vue';
import * as Sentry from '@sentry/browser';
import * as Integrations from '@sentry/integrations';

Sentry.init({
	dsn: 'http://1b036df22e114db78f99770cf9be29b0@localhost:9000/3',
	integrations: [new Integrations.Vue({ Vue, attachProps: true })],
});

到这里就完成了 nodejs 和 vuejs 的异常捕获以及上报

因为 vue 项目大多是使用 webpack 构建出来的，代码已经被压缩混淆了，所以上线后的异常是没办法精确到具体行号，如果需要精确的错误信息，请看下面

# 更进一步

# 区分开发环境与生产环境

我们真正关注的其实只有生产环境的异常，对开发环境的异常我们都是立马解决的，所以对于异常监控，我们一般都需要对环境进行区分，下面有两种方式

  1. 生产环境才上报

这是最简单也是最方便的，只在生产环境下时才上报异常，通过process.env.NODE_ENV判断即可

// 初始化nodejs版错误监控服务
if (process.env.NODE_ENV === 'production') {
	Sentry.init({ dsn: nodeDsn });
}
  1. 两套异常项目

如果也需要监控开发环境的（当然得和生产环境的异常分开），我们一般是通过创建两套 sentry 项目，分别监控

// 初始化nodejs版错误监控服务
Sentry.init({ dsn: process.env.NODE_ENV === 'production' ? nodePordDsn : nodeDevDsn });

# 上传 source map 精确定位异常

前面提到，vue 项目（包括其他用 webpack 构建出来的项目），监听到的异常并不完整，难以精确到具体代码，所以我们需要上传 source map 文件到 sentry

上传 source ma 前

上传 source map 后

很明显，上传 source map 后 Sentry 定位的更加精确了

下面介绍如何上传 webpack 项目构建的 source map

# 选择上传方式

上传方式有两种，分别是通过@sentry/cli脚手架人工上传和通过webpack插件自动上传

链接：@sentry/cli 、 @sentry/webpack-plugin

既然我们是使用 webpack 构建的，选用 webpack 插件 自然是最合适的了

# 生成 source map

在 webpack 中，devtool属性有四个值，可以配置生成映射文件帮助我们调试代码

devtool
source-map
eval-source-map
cheap-module-source-map
cheap-module-eval-source-map

这里选用 source-map 即可

# 生成 sentry 配置文件

既然需要 webpack 中上传 map 映射文件，那么就需要明确说明需要上传到哪个服务器、哪个项目下

@sentry/webpack-plugin插件会自动读取项目根目录的 .sentryclirc 文件

  1. 首先我们在项目根目录创建 .sentryclirc 文件

  2. 再去 sentry 网站，侧边栏 / 用户 / API keys / Auth Tokens ，创建一个 token

这时候需要勾选project:write，生成后进会得到一段 token 字符串，这是将来上传 map 文件的凭证

  1. 确定项目名称，需要知道 sentry 上对应的 dsn 的那个项目名称

  2. 确定组织名称，在 Settings / General 中可以看到

默认是 sentry

  1. 编写.sentryclirc
[defaults]
url = 你部署的sentry地址
org = 前面拿到在组织名
project = 前面拿到是项目名（不是代码项目，是sentry服务上的项目名）

[auth]
token = 前面拿到的token字符串

# 实例化插件

配置好信息后，我们可以做自动上传部分了

安装依赖

$ npm install @sentry/webpack-plugin --save-dev

实例化

// prod.config.js
const path = require('path');
const SentryPlugin = require('@sentry/webpack-plugin');

plugins[
  ...
  new SentryPlugin({
        // map文件所在文件夹
        include: path.resolve(__dirname, '../dist'),
  }),
  ...
]

这时候再执行 npm run build 构建命令，就会自动上传 map 文件了

# 注意事项

  1. 构建错误
 ERROR in Sentry CLI Plugin: Command failed: E:\xxx\xxx\node_modules\@sentry\cli\sentry-cli.exe releases new claimform@1.0.3

error: project not found

Add --log-level=[info|debug] or export SENTRY_LOG_LEVEL=[info|debug] to see more output.
Please attach the full debug log to all bug reports.

这往往是你的 1.sentryclirc 配错了，检查一下 org 和 project 信息

  1. 配置错误
ERROR in Sentry CLI Plugin: Command failed: E:\xxx\xxx\node_modules\@sentry\cli\sentry-cli.exe releases new claimform@1.0.3

error: project not found

检查一下 sentry 配置文件名是否写错 .sentryclirc

  1. map 错误
~/static/js/app.xxxxxxxxxxx.js (no sourcemap ref)
- warning: could not determine a source map reference (Could not auto-detect referenced sourcemap for ~/static/js/app.xxxxxxxxx.js.)

检查一下 devtool 是否开启了 source map

  1. http status 413
sentry reported an error: unknown error (http status: 413)

文件上传大小超过限制，检查一下是否 sentry 服务是否用了反向代理，如 nginx，修改文件大小限制

$ vim /etc/nginx/nginx.conf

在 http 字段下修改

client_max_body_size 20M; # 修改到20M或更大

重启 nginx

$ service nginx restart
