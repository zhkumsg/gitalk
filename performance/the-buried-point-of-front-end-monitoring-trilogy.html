# 前端监控三部曲之数据埋点

获取用户基本信息、行为以及跟踪产品在用户端的使用情况，并以监控数据为基础，指明产品优化的方向

三部曲：数据埋点、性能监控、异常监控

# 主要思路

1. （何人）什么人

2. （何时）在什么时候

3. （何地）在什么地方

4. （How）通过什么方式

5. （何事）干了什么事

# 前端埋点方式

方式 优点 缺点
代码埋点 可自定义属性，自定义事件/可以细化需求/相比其他埋点方式减少服务器压力（使用 vue 指令） 工程量大的话，手动埋点会出现疏漏，不方便审查 / 需求变更要重新埋点，成本高 / 每次需求变更都要重新发布版本，对线上系统稳定性有一定危害
全埋点 前端只要一次加载埋点脚本 服务器性能压力大
可视化埋点 前端只要一次加载埋点脚本 ,可视化配置 服务器性能压力大

# 代码埋点

# 手动埋点

命令埋点，前端代码中需要监控的地方插入监控逻辑

// 页面加载时发送埋点请求
$(document).ready(function() {
	// ... 这里存在一些业务逻辑
	sendRequest(params);
});
// 按钮点击时发送埋点请求
$('button').click(function() {
	//  这里存在一些业务逻辑
	sendRequest(params);
});

# 声明式埋点

声明式埋点的思路是将埋点代码和具体的交互和业务逻辑解耦，开发者只用关心需要埋点的控件，并且为这些控件声明需要的埋点数据即可，从而降低埋点的成本 ，在 dom 元素上增添埋点信息

<!-- key表示埋点的唯一标识；act表示埋点方式 -->
<button data-stat="{key:'111', act: 'click'}">埋点</button>

优点：可以在任意时刻，精确的发送或保存所需要的数据信息，让使用者可以方便地设置自定义属性、自定义事件（对前端框架 Vue 使用指令属性可快速封装）；

缺点：项目工程量大，需要埋点的位置太多，而且需要产品开发运营之间相互反复沟通，容易出现手动差错，如果错误，重新埋点的成本也很高。这会导致整个数据收集周期变的很长，收集成本变的很高，而且效率很低。因为手动埋点需要开发人员完成，所以每次有埋点更新，或者漏埋点，都需要重新走上线发布流程，更新成本也高，对线上系统稳定性也有一定危害；

# 全埋点/无埋点/自动埋点

又叫无埋点，无埋点并不是没有任何埋点，所谓无只是不需要工程师在业务代码里面插入侵入式的代码。只需要简单的加载了一段定义好的 SDK 代码，技术门槛更低，使用与部署也简单，避免了需求变更，埋点错误导致的重新埋点,

优点：不需要关注埋点逻辑的，也不会出现漏埋、误埋等现象，前端只要一次加载埋点脚本，解决数据 回溯 问题上更有优势；

缺点：自定义属性不灵活，传输时效性差，数据可靠性欠佳，耗费网络流量，还会增加服务器负载，而且兼容性也不佳；

# 可视化埋点/无痕埋点

运营通过可视化的界面拖拽配置实现，这些活动控件元素都带有唯一标识。通过埋点配置后台，将元素与要采集事件关联起来，可以自动生成埋点代码嵌入到页面中

业内开源解决方案：Mixpanel

可视化埋点配置化能力相对手动埋点更强，是对手动埋点的补充而不是代替，很多手动埋点点都可以通过好的规划和架构变为可视化埋点

在实际项目中考虑到上报数据的灵活定制，以及减少数据传输和服务器的压力，在所需埋点处不多的情况下，常用的方式是代码埋点

# 用户模型

# 访问用户

访问用户是应用（包括网页、App、微信小程序等，下同）的用户的一种识别机制。

每一个访问应用的用户都会在对应的设备中生成并记录一个唯一的 ID，称之为访问用户 ID。

# 登录用户

登录用户是访问用户绑定到业务系统账号的一个机制，类似于单点登录后的用户
